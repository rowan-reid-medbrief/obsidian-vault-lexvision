This project directory (`~/claude_code/medbrief`) is a real git clone of MedBrief Secure Review, git identity `rowan.reid@medbrief.co.uk`. **Origin migrated Bitbucket‚ÜíGitHub Enterprise on 2026-07-09** (the team-wide move; guide = Confluence [page 1056079874](https://medbrief.atlassian.net/wiki/spaces/MDC/pages/1056079874) section 2): `origin` is now `git@github.com:medbrief/msr-medbrief.git` (was `git@bitbucket.org:medbriefteam/medbrief.git`). GitHub SSH authenticates as `rowan-reid-medbrief`; new-branch pushes are accepted (Bitbucket had blocked them). Repo naming prefix convention: `msr-` = MedBrief Secure Review (see the guide's section 4 table).

It is live-synced via **Mutagen** (session name `medbrief`) to Rowan's own DigitalOcean dev VM, where the actual docker stack runs (`app`, `php`, `db`, `db_test`, `mailcatcher` ‚Äî confirmed up continuously, 6+ weeks uptime as of 2026-07-03). The sync excludes `.git`, `vendor`, `node_modules`, `var`, `tmp`, `data`, `public/build`, `public/bundles`, `public/app`, `storybook-static` (~1.3GB of build artefacts/dependencies that don't need to cross the wire). See [[medbrief-dev-vm-access]] for connection details.

**Why:** Rowan wanted to edit MedBrief code with Claude Code locally without SSHing into the VM by hand for every edit. Mutagen was chosen over SSHFS (the VM's `vendor`/`node_modules` are 1.3GB ‚Äî too slow over a real network filesystem) and over a plain git-clone-and-push cycle (loses live edit-and-see-it-running feedback). Edits made locally reflect on the VM within a couple of seconds; the running containers bind-mount the whole repo (`../:/var/www/html:delegated`) so no rebuild is needed.

**How to apply:** Because `.git` is excluded from sync, all git operations (branch, commit, push) should happen on this Mac side, not on the VM directly ‚Äî the VM's own `.git` won't know about branches created here. Check sync health with `mutagen sync list`; pause/resume with `mutagen sync pause/resume medbrief`. Note: this Mac's Homebrew has tap-trust enabled, and the auto-mode classifier blocks Claude from running `brew trust` itself even with user approval ‚Äî ask Rowan to run trust commands himself when a new tap is needed.

**Permissions gotcha ‚Äî fixed permanently on 2026-07-04.** Mutagen previously synced files into the VM as `0600`/dirs `0700`, root-owned, which php-fpm (`www-data`) couldn't read ‚Äî new/edited PHP classes and config (e.g. `config/services/interface_aliases.yaml`) threw "file cannot be read" / `LoaderLoadException` on page load. Root-run CLI (`bin/console`, PHPUnit) always worked fine; only the browser-facing app broke.

**Fix applied:** recreated the `medbrief` Mutagen sync session with `--permissions-mode=manual --default-file-mode-beta=0644 --default-directory-mode-beta=0755`, so every file/dir Mutagen transfers to the VM from now on lands already world-readable ‚Äî no more per-file chmod firefighting. `.env*` was deliberately added to the sync ignore list (alongside the existing `.git`, `vendor`, `node_modules`, etc.) so this permissions loosening never touches secrets; the VM keeps its own static copy of `.env`/`.env.dev`/`.env.local`/`.env.test`, which no longer auto-updates from the Mac side.

**Caveat:** the manual mode only normalizes permissions on files Mutagen actually re-transfers. Files that already matched byte-for-byte between Mac and VM before the recreate kept their old `0600`/dir `0700-with-o+rx` state and needed a one-time manual `chmod o+r`/`o+rx` catch-up (done for the branch in flight at the time: `ProjectDocumentController.php`, `MedicalRecord.php`, `RecordsViewer` JS files, `project-document.yml`, `Mri134SeedDemoCommand.php`, `codingStandards.md`, `PDFTools/PageSpec`, `PDFTools/Xod`). If an old file from before this date still throws this error, it likely never got touched by a transfer since the recreate ‚Äî one-off `ssh 10.40.87.204 "chmod o+r <path>"` plus `docker restart docker-php-1` clears it.

Check current sync config any time with `mutagen sync list medbrief --long`.

**Login gotcha (found 2026-07-04, not fixed at the code level).** The seeded `super-admin@medbrief.dev` account (`fos_user` id 2) was created disabled (`enabled=0`) by a bug in `src/DataFixtures/UserFixtures.php`'s `USER_SUPER_ADMIN` fixture, which alone among its sibling admin fixtures is missing `->setEnabled(true)`. Confirmed via a throwaway Symfony command using the app's own password encoder that `password` is the correct stored password and the account is simply switched off, not a wrong-credential issue. Worked around with a direct one-row `UPDATE fos_user SET enabled=1 WHERE id=2` on the dev DB. Rowan declined to fix the fixture itself, so a future `bin/console doctrine:fixtures:load` (or `dev-only-reset-db.sh`) will recreate the same disabled state ‚Äî re-apply the same UPDATE if login fails again after a fixture reload. See [[medbrief-avoid-broad-chmod-credential-checks]] for how this was diagnosed without printing the password hash.
