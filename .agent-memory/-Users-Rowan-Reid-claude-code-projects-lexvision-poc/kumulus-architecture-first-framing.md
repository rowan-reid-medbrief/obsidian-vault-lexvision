Rowan's position on the Kumulus engagement (stated 2026-06-10, when the revised proposal inverted the scope split): the primary artefact is the modular architecture definition, meaning what the project looks like, what the modules are, how each is built, and how each is measured (proper benchmarks and approaches per module). Who builds each module (Kumulus or MedBrief) is fluid and secondary. Module-level open questions stay live regardless of the vendor split; they are addressed to whichever side owns the module at the time, and asking Kumulus is secondary to agreeing what a good version of the architecture looks like.

**Why:** the proposal scope flipped twice in two weeks (28 May offered to drop the comparison layer; 10 June dropped everything except it), so any document that anchors questions to "what Kumulus must answer" goes stale on every scope change, while the architecture framework does not.

**How to apply:** when writing or amending dossier docs, deliverables, or revision notes, do not frame module questions as superseded because the vendor split changed. Frame them as live project questions with fluid ownership. This matches dossier doc 05 open question 5 (the MedBrief-authored modular framework as the engagement anchor) and the d1 modular map.
