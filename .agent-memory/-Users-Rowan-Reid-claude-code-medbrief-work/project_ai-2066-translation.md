Azure DevOps work item **2066** (MedBrief AI project), a Compre translation and pseudonymisation workflow spike assigned to Rowan. Full working detail in `docs/ai-2066/`; vault thread in `Next Up.md` under `## Now (Medbrief)`.

Five non-obvious findings, expensive to re-derive:

- **Residency: a uksouth Translator resource processes in France Central / West Europe, NOT the UK** (found 2026-07-16 in Microsoft's "Document translation data residency" table; `uksouth` falls under the "Europe (except Switzerland)" row). This **corrected the workstream's own earlier claim** that the translator was "UK-resident, which resolves the estate/residency concern" and that "region confirmed UK". The recorded bar was "UK **or** EU" so the spike is not blocked, but production is a UK-at-rest to EU-processing transfer needing DPA sign-off (for Deon, not engineering). Unverified lever: a gated-access Translator Docker container would process in-estate. **If any note or person says "region confirmed UK", it is wrong.**
- **Azure silently half-translates mixed digital/scanned PDFs.** The FAQ states it plainly: mixed documents get "only the digital portions translated", with no error raised. Medical records are exactly that shape (digital cover letter plus scanned notes), so the output is a plausible-looking artifact with untranslated content, which is worse than a crash for a medico-legal product. The FAQ's own mitigation is rasterising to fully-scanned before submission. Whether "mixed" means within-page or across-pages is unresolved and determines a pipeline branch; only the hands-on run settles it.
- **Azure Document Translation refuses PDF on the synchronous endpoint** (2026-07-10 hands-on test; `document:translate` returns `InvalidFormat/ContentType` on every api-version); it accepts text, Office and HTML only. PDF is **batch-only** (`document/batches`), staging source and target in **blob containers**. So the translation pass is inherently an async, storage-backed step (the `salli_func` Azure-Function precedent), not an inline call. **Evidence asymmetry worth knowing** (checked 2026-07-16): the blob requirement IS explicitly stated in Microsoft's docs (three places, verbatim), but the PDF refusal is **not stated anywhere**; the only documentary trace is PDF's absence from the Synchronous tab of the overview's format table. Rowan's own test is the stronger citation, so cite it rather than the vendor docs.
- **Azure access, the operative map** (full-estate check 2026-07-16; supersedes a mis-scoped 2026-07-15 finding that read as "Rowan cannot self-serve storage" full stop, when it only ever checked Azure DevTest). Rowan's four subscriptions: **Azure DevTest** (`65483d0a-…`, holds `clinicalrecord-translator`) gives him four resource-scoped roles and **no create rights**; **Foundation A.I.** and **MedBrief Analytics** give him **nothing**; **MedBrief Secure Review** (`7b66abb8-…`, production) gives him **Contributor at subscription root, unconditional**. So **he can create storage accounts and Cognitive Services resources in production himself** (`Microsoft.Storage` and `Microsoft.CognitiveServices` both Registered; no Deny policies, only Defender/ASC monitoring). He also holds **Owner** on `stgsalli3odev` (DevTest, Salli's account). Earlier notes also had the translator's subscription ID wrong, recording Secure Review's; corrected 2026-07-15.
- **The one ceiling everywhere: Rowan cannot create role assignments.** Contributor carries `notActions: Microsoft.Authorization/*/Write`. So the **SAS path is fully self-serve** (Contributor's `actions: *` includes `listKeys`, and the harness already uses SAS, so **the spike needs nobody's permission**), but the **managed-identity path is blocked** and needs Owner/User Access Administrator to grant the Translator's identity blob access. That is the only real permission ask in the whole design, and it is a production concern, not a spike one. Exception: on `stgsalli3odev` he has Owner and could wire MI there.
- **So "Rowan is blocked on storage" was always wrong**, and the staging decision paused since 2026-07-10 is **a governance question, not a permissions one**: should spike resources holding PHI be created in the production subscription? His to raise, not to be granted. The blocker was an artefact of assuming staging had to sit beside the DevTest translator; SAS URLs are subscription-agnostic, so Secure Review storage works with the DevTest translator today. Secure Review is also where the PHI already lives (`msrstandard`), where `comprepseudo` already sits, and where the design said production belongs anyway.

**The reframe worth remembering** (raised 2026-07-16, unanswered): the translated-PDF premise is what forces the whole batch → blob → staging → cross-border → mixed-document chain. MedBrief already OCRs every document, and the sync API takes text natively with none of that machinery. AC5's "existing English translated pages **alongside** the German" suggests the current human workflow produces parallel pages, not an in-place PDF swap, so the premise may be dissolvable. Cheap to ask Deon, expensive to discover late.

**State at 2026-07-16 session end.** No batch run executed, no PHI has left the machine. The spike is unblocked technically and waits only on a governance answer. A Teams message to Deon and Laura is **drafted and unsent** at `docs/ai-2066/DRAFT-teams-message.md`: it reports the batch-only finding, states Rowan will create staging in Secure Review himself, and asks the two things he cannot answer alone (is there an intended harness already? and does the FR/NL processing need a DPA check?). **A methodological lesson worth carrying**: twice in this workstream a confident recorded claim turned out to be wrong on checking, and both were load-bearing ("region confirmed UK"; "Rowan cannot self-serve storage"). Both came from checking one thing and generalising. When a claim in these notes blocks or unblocks work, re-derive it rather than inheriting it.
