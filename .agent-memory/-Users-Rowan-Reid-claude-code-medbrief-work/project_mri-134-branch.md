**Follow-on session (2026-07-10): the "19x MedBrief overhead" did not exist; the copy was the villain; cap removed. Two commits pushed.** Profiling the extraction path (the previous session's next step) overturned its own premise. Measured warm and back-to-back, the booted kernel runs the identical PDFNet operation at the same speed as standalone PHP: the "~12-22x runtime overhead" was a **page-cache confound**. `copyToLocal` read and rewrote the whole multi-GB source just to extract a few pages, doubling the working set past the 8GB box's RAM, so the extract then read cold and slow. **This supersedes the 2026-07-09 conclusion above that `copyToLocal` is "production-correct... leave as-is".** Fix: `FileHandlerInterface::resolveLocalSourcePath()` (+ Vich impl) returns a plain local file's real path so the source is opened in place, with `copyToLocal` kept only for blob/Azure sources, and the temp-unlink guarded by an ownership flag. Committed `3a0b5d109`. At 10,000 pages: 89s -> 46s warm / 53s cold; at 500 pages: 26s -> 1.8s. The saving is architectural (no copy, no working-set doubling), so it holds cold as well as warm.

Second commit `27547ede3`: **the 500-page cap was ours, not inherited** (`PageSpecParser.php` and `streamPageSubsetAction` are absent from `origin/develop`), and its docblock rationale was half-wrong: `assertWithinPageCount()` already refused any page past the document's end, so the cap never guarded extraction. Its one real job was stopping the parser expanding `1-999999999` into a billion-element array before the page count was known. So the number was dissolved rather than raised: `parseRanges()` measures ranges arithmetically, `assertWithinPageCount()` bounds them against the open document, `expandRanges()` runs last. A legitimate 9,000-page date range now parses in 4.9ms; `1-999999999` is refused in 0.058ms with a zero-byte memory delta. What survives is `SANITY_MAX_PAGES = 1_000_000` plus a `@todo` for a config-bound product limit. Both commits are on `origin/feature/MRI-134-pdf-split-approach`.

**Engine question settled for Deon:** PHP-Apryse-11.4 and Python-Apryse-12.0 are the same speed at every size (same C++ core), so Python is not faster than PHP; the earlier contrary claim was a warm-vs-cold confound. PyMuPDF is disqualified: 133s at 10,000 pages (~2.8x Apryse), 2.9GB RSS, and it cannot linearise, so its output is not viewer-streamable.

**Three constraints found, all pre-existing, none of them the cap:** (1) the PHP request wall is nginx's **60s fastcgi default**, not 600s (the `proxy_read_timeout 600` lines cover the mock and radiology proxies; the PHP location is `fastcgi_pass` with no `fastcgi_read_timeout`), so a 10k-page extract at 47-53s brushes it; (2) php-fpm runs `pm.max_children = 5` with `request_terminate_timeout = 0s`, so a long extract pins a worker and nothing reclaims it, which is the real argument against simply raising the timeout; (3) the subset cache under `sys_get_temp_dir()` has no TTL or size bound (~2.9GB per 10k subset). Logged as ideas in `docs/mri-134/IDEAS.md`. Verified by 23 unit tests, PHPStan and Psalm; the functional controller test still cannot run because of the `MatchLoA` test-container failure recorded above, confirmed pre-existing by reproducing it at `f0314aea1`.
