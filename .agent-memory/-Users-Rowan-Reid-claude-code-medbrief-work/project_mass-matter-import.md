Mass Matter Import: a one-off bulk import of the client **Higgs**' matters and radiology/hospital records into MedBrief, briefed by Deon Kuhn to Rowan on 2026-07-22. Build it as a Symfony console command IN the MedBrief product codebase (`repo/`), not standalone Python: matter creation means creating Doctrine entities, and the "processing queue" is just `Disc.status` polled by cron.

**No tracker ticket exists yet** (checked Jira + Azure DevOps 2026-07-22). Docs live under `docs/mass-matter-import/` with a provisional slug; rename if a ticket is cut.

**HARDENED 2026-07-23 (14 Claude critics + 11 Copilot lens legs). THREE PREMISES THIS PROJECT WAS BUILT ON TURNED OUT FALSE.** Full consolidated findings in `~/.claude/plans/.harden-runs/mass-matter-import/CONVERGENCE.md`; corrections in DESIGN-NOTES §16. The lesson, now recorded in the gh-copilot ledger and ideas: every claim accepted by checking that the cited *lines* said what was claimed was false; every one re-verified by reading the surrounding *control flow* held.

Load-bearing findings:
- ~~`Project::setManager()` grants NO authorisation~~ **FALSE.** `setManager()` alone gives the fee earner BOTH record-tree access (`ProjectVoter:411-422`, the enclosing `if != DELETE && != ADMINISTRATION` was missed on the first read) AND listing visibility (`UserRestrictableQueriesTrait:180-183`, "Always show projects where the user is the Manager"). §10.3 "the most consequential finding of the day" was not a finding. **Recommendation: drop the role grant entirely** (§16.1a) since it is the only write touching pre-existing shared rows and cannot be undone; Rowan's call as it was a ruling. What DOES matter now: scope the fee-earner lookup to account 296 and hard-stop outside it (§16.2, ruled), the only remaining guard against a wrong-firm user.
- ~~Create four Collections~~ **FALSE: create ONE.** `Project::__construct()` already makes Records + Unsorted Records (cascade all); the live `MatterBuilder` adds only Private. Copying the deprecated `createAction` orphans three rows per matter (§16.6, verified by hand).
- ~~No import counterpart to build on~~ **FALSE.** The search globbed `*MatterImport*`; the files are `ImportMatters*`. A routed `ImportMattersController`, a broken-in-prod `ProjectFactory`, and a non-deprecated `ImportRadiologyCommand` all exist. Reuse matrix in `mmi-reuse-decision.task` output.
- Five mainline-breaking defects (§16.5): `wrapInTransaction` CLOSES the EntityManager on failure (so "log and continue" cascade-fails the run); `processArchive()` returns bool not throws (commits a matter with no records); no MD5 dedupe exists; virus scanning is backwards (records need it, discs get it free); no DB-level duplicate constraint.
- **NEAR-TERM TARGET IS LOCAL** (ruled 2026-07-23, §16.4): a local dry run + a local `--commit` run with discs left at `PENDING_PROCESSING` (cron not running). Async processing out of scope. Staging/pre-staging/production stays an open Deon decision. The current build plan is `docs/mass-matter-import/LOCAL-BUILD.md`, NOT the long-form PLAN.md (which is a hybrid pending redraw).
- **Design now centres on a first-class dry-run + declarative integrity-check framework** (DESIGN-NOTES §12, settled with Rowan 2026-07-23): dry-run is the default (writes nothing), `--commit`/`--execute` to persist; checks are `{id, description, test, severity}` with severity in one config table (trivial warn↔hard-stop reclassification, Deon may promote some). Phase 1 builds the runner + all read-only checks (the account-manager cross-check `fee_earner.is_account_manager` is the first, a diagnostic not a gate); Phase 2 is the `--commit` write stage reusing the runner. Gotcha: dry-run validates the estate it runs against, so run it on the same env you commit to.
- **Fixtures must be informed by the real Higgs structure, not invented** (Rowan, 2026-07-23; DESIGN-NOTES §14). Phase 1 now splits: 1a framework machinery (local, unblocked), 1b a VM-only structural-profiling step that emits a PHI-free structural profile (exact column headers, real naming conventions, the valid/malformed boundary), 1c fixtures generated from that profile (incl. deliberately-malformed cases) + checks + pass/fail self-test. Trust dependency: trustworthy pass/fail fixtures need 1b OR Deon's anonymised source; until then a passing run proves the framework only, not correctness against the real shape. **The one genuinely irreversible risk is PHI leaking off the VM in the profile** - redact/synthesise at the point of capture on the VM, review on the VM before it comes local (D7).
- **`Project.account` is non-nullable** and a wrong value causes cross-tenant exposure. RESOLVED 2026-07-23: the account is **Higgs LLP, account_id 296** (slug `higgs_llp`), confirmed by direct query of the pre-staging DB (replica of prod, port 3310 via jump-box tunnel; creds in workspace `.env`). Pass it as a command param (resolve the slug at runtime + fail closed on non-unique match), NOT a sheet column, since one account covers the whole import (§9.1, §10.1).
- Real Higgs data may run on Rowan's DigitalOcean VM (over ZeroTier), NEVER on local. Deon's binding rule (see [[project_medbrief_dev-environment]]).
